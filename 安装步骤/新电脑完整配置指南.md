## 1. 项目简介

| 项目 | 说明 |
|------|------|
| **技术栈** | Vite 6 前端 + Node.js API（Hono）+ SQLite |
| **管理端** | 编辑表格、SVG 编辑框、导出 PDF/SVG、证书草稿/发布 |
| **公众端** | 只读浏览已发布证书 |
| **默认开发端口** | 前端 `5173`，API `3001` |

开发时通常同时启动 **API** 与 **Vite**；前端通过代理访问 `/api` 和 `/uploads`。

## 2. 环境要求

### 2.1 操作系统

- **Windows 10/11**（当前项目主要在 Windows 上开发）
- 也可在 macOS / Linux 上运行，命令基本相同（路径、脚本名略有差异）

### 2.2 Node.js（必须）

| 要求 | 说明 |
|------|------|
| **推荐版本** | Node.js **20.19+** 或 **22.12+**（见项目 `.nvmrc` 为 `22`） |
| **最低** | `package.json` 写明 `>=20.19.0`；低于此版本 Vite 6 会报错 |
| **下载** | https://nodejs.org/ （选 LTS） |

安装后在 **新终端** 验证：

```powershell
node -v
npm -v
```

应看到类似 `v22.x.x` 或 `v20.19.x`。

> **注意**：若 `node -v` 仍是旧版本，请把 PATH 中 Node 路径改为官方安装目录，例如 `C:\Program Files\nodejs\`。

### 2.3 其它软件

| 软件 | 是否必须 | 用途 |
|------|----------|------|
| **Git** | 可选 | 若用仓库克隆而非 zip |
| **Visual Studio Build Tools** | 仅当 `npm install` 编译失败时 | 编译 `better-sqlite3` 原生模块 |
| **7-Zip / 系统解压** | 解压备份 zip | — |

### 2.4 网络

- 开发时字体可从 **CDN** 或 **本机路径** 加载；若字体源填外网 URL，需能访问该地址。
- `npm install` 需能访问 npm  registry。

## 3. 需要拷贝哪些文件

### 3.1 完整迁移（保留全部数据与依赖）

从旧电脑备份的 zip（例如 `Cat2_backup_2026-05-28_021627.zip`）解压后，应包含整个 `Cat2` 目录。

**务必保留或单独拷贝：**

| 路径 | 说明 |
|------|------|
| `data/cat.db` | **核心数据库**：管理员、证书、表格行、SVG 模板、布局预设、字体源配置等 |
| `data/uploads/` | 单元格图片等媒体（若使用过图片列） |
| `.env` | API 端口、管理员密码、JWT 密钥、CORS |
| `layout-settings.json` | 项目根目录编辑框布局备份 |
| `public/layout-settings.json` | 与上一项同步（开发时 Vite 读取） |
| `src/default-layout-settings.json` | 「设为默认」写入的出厂布局（若有） |

### 3.2 可不拷贝（到新电脑重新生成）

| 路径 | 说明 |
|------|------|
| `node_modules/` | 换电脑或换 CPU 架构后建议 **删除并重新 `npm install`** |
| `dist/` | 执行 `npm run build` 可重新生成 |
| `node_modules/.vite/` | Vite 缓存，可删 |

### 3.3 敏感文件

- **不要** 把 `.env` 上传到公开网盘或发给无关人员。
- 新电脑若不用旧密码，可在 `.env` 改 `ADMIN_PASSWORD` 后执行 `npm run reset:admin`（见下文）。

## 4. 方式 A：从完整 zip 备份恢复（推荐）

适用于：已有完整项目 zip（含源码、`data`、`.env` 等）。

### 步骤 1：安装 Node.js

1. 打开 https://nodejs.org/ 下载并安装 **LTS（20 或 22）**。
2. 安装时勾选「添加到 PATH」。
3. 关闭并重新打开 PowerShell 或 CMD，执行 `node -v` 确认版本。

### 步骤 2：解压项目

1. 将 zip 解压到目标位置，例如：
   - `D:\Projects\Cat2`
   - 或 `C:\Users\你的用户名\Desktop\app\Cat2`
2. 确认目录内能看到：`package.json`、`server/`、`src/`、`index.html` 等。

### 步骤 3：处理依赖（重要）

若 zip **来自另一台电脑** 或 **CPU 架构不同**（如 Intel ↔ ARM），请：

```powershell
cd D:\Projects\Cat2   # 改成你的实际路径

# 删除旧依赖（避免 better-sqlite3 二进制不匹配）
Remove-Item -Recurse -Force node_modules -ErrorAction SilentlyContinue

npm install
```

若 zip 是 **同一台电脑同架构** 的完整备份且 `node_modules` 完好，可尝试直接启动；若报 `better-sqlite3` 相关错误，仍按上面删除后重装。

### 步骤 4：检查 `.env`

若解压后 **没有** `.env`：

```powershell
copy .env.example .env
notepad .env
```

至少修改：

- `JWT_SECRET` → 随机长字符串
- `ADMIN_PASSWORD` → 你的管理员密码

若从旧机拷贝了 `.env`，一般 **无需修改** 即可登录（用户名默认 `admin`）。

### 步骤 5：确认数据目录

确认存在：

- `data/cat.db`（有数据时体积通常 > 0）
- 可选：`data/.installed`（表示已走过安装流程，本地开发正常）
- 可选：`data/uploads/`（有上传图片时）

若没有 `data/cat.db`，首次启动 API 会创建空库，需重新登录并在后台导入模板/证书。

### 步骤 6：启动

```powershell
cd D:\Projects\Cat2
npm run dev:local
```

- `dev:local` 会：检查 Node 版本 → 创建/确认 `data/.installed` 与 `.env` → 结束占用 3001 的旧 API → 同时启动 API + Vite。
- 浏览器一般自动打开；若无，手动访问下面 [页面与地址](#8-页面与地址一览)。

### 步骤 7：验证登录与数据

1. 打开 http://localhost:5173/login.html  
2. 使用 `.env` 中的 `ADMIN_USERNAME` / `ADMIN_PASSWORD` 登录（默认用户名为 `admin`）。  
3. 进入编辑页，确认证书列表、SVG 模板、编辑框布局是否正常。  
4. 打开 **设置 → 字体源**，确认已启用字体仍在；在编辑框布局栏测试「字体」下拉。

## 5. 方式 B：仅拷贝源码后全新安装

适用于：只拷贝 git 仓库或不含 `node_modules` / `data` 的压缩包。

### 步骤 1～2

同 [方式 A](#4-方式-a从完整-zip-备份恢复推荐) 安装 Node.js 并解压/克隆到目标目录。

### 步骤 3：安装依赖

```powershell
cd D:\Projects\Cat2
npm install
```

### 步骤 4：从旧电脑单独拷贝数据

将旧电脑的以下文件复制到新项目对应位置：

```
data/cat.db
data/uploads/          （整个文件夹，若有）
.env                   （或新建后改密码）
layout-settings.json
public/layout-settings.json
```

### 步骤 5：初始化本地开发标记

```powershell
npm run dev:local
```

首次运行会自动：

- 从 `.env.example` 复制 `.env`（若不存在）
- 创建 `data/.installed`

### 步骤 6：构建前端（仅生产或不用 Vite 时需要）

```powershell
npm run build
```

开发日常 **不需要** 先 build，直接用 `npm run dev:local` 即可。

## 6. 配置环境变量 `.env`

项目根目录 `.env` 示例（见 `.env.example`）：

```env
# API 服务端口
PORT=3001

# 会话签名密钥（生产必须改为随机长串）
JWT_SECRET=请改为随机长字符串

# 管理员账号（首次无用户时由 API 创建）
ADMIN_USERNAME=admin
ADMIN_PASSWORD=你的强密码

# 开发时前端地址（CORS）
CORS_ORIGIN=http://localhost:5173

# 生产部署时取消注释
# NODE_ENV=production
```

| 变量 | 说明 |
|------|------|
| `PORT` | API 监听端口，默认 `3001` |
| `JWT_SECRET` | Cookie 会话签名，**每台环境建议不同** |
| `ADMIN_USERNAME` / `ADMIN_PASSWORD` | 管理员登录；改密码后可用 `npm run reset:admin` 同步到数据库 |
| `CORS_ORIGIN` | 开发时为 `http://localhost:5173`；生产改为实际域名 |

### 重置管理员密码

修改 `.env` 中的 `ADMIN_PASSWORD` 后执行：

```powershell
npm run reset:admin
```

按提示使用新密码登录。

## 7. 启动开发环境

### 7.1 推荐命令（Windows）

```powershell
cd 项目根目录
npm run dev:local
```

等价于：检查 Node → 标记已安装 → 清理旧 API 进程 → `npm run dev`（API + Vite 并行）。

### 7.2 其它脚本

| 命令 | 说明 |
|------|------|
| `npm run dev` | 同时启动 API(3001) + Vite(5173) |
| `npm run dev:web` | 仅前端（需另开终端跑 API） |
| `npm run dev:server` | 仅 API |
| `npm run dev:win` | 调用 `scripts\dev-win.cmd`，优先使用 `C:\Program Files\nodejs` |
| `npm run check:api` | 检测 3001 API 是否为当前版本 |
| `npm run build` | 构建 `dist/` |
| `npm start` | 仅启动 API（生产或已 build 后） |
| `npm run start:prod` | `build` + 启动 API |

### 7.3 停止服务

在运行 `dev:local` 的终端按 **Ctrl + C**。

### 7.4 端口被占用

若提示 3001 或 5173 已被占用：

```powershell
# dev:local 会尝试结束旧 API；仍失败时可手动查端口
netstat -ano | findstr :3001
netstat -ano | findstr :5173
```

结束对应 PID 后重新 `npm run dev:local`。

## 9. 业务数据与配置文件说明

### 9.1 SQLite 数据库 `data/cat.db`

存储内容包括（不完全列表）：

- 管理员账号 `admin_user`
- SVG 模板 `svg_templates`
- **表格模板** `table_templates`（列名与列顺序，如「标准血统表」）
- 布局预设 `layout_presets` 及修订
- 证书 `certificates`（含 **`column_order` 列顺序**）、表格行 `certificate_rows`、修订历史
- 字体源配置（键值存储在 settings 表）

**迁移时最重要**：务必拷贝此文件，否则证书与模板全丢。

> **关于 `cat.db-wal` / `cat.db-shm`（常见误区）**  
> 这两个是 SQLite **WAL 模式下的运行时临时文件**，只在程序运行期间存在。  
> - **迁移/恢复时不需要** 从旧电脑单独拷贝它们。  
> - **切勿** 把旧机的 `cat.db-wal` 和新机的 `cat.db` 混用——会导致数据错乱或看起来「丢数据」。  
> - 恢复时：只替换 `cat.db`，并**删除**同目录下的 `cat.db-shm`、`cat.db-wal`，再重启 API。  
> - 若程序正在运行就拷贝，可能只拷到不完整的 `cat.db`（未合并 WAL 中的最新写入）。**应先停 API**，或使用下文 `npm run backup:data`（自动合并 WAL）。

#### 备份数据（推荐）

**方式 A：后台界面（超级管理员）**

登录后台 → **设置 → 数据维护** →「创建备份并下载」。文件会保存到 `data/backups/` 并触发浏览器下载。

**方式 B：命令行**

```powershell
npm run backup:data
```

- 输出文件：`data/backups/cat-backup-YYYY-MM-DD_HHmmss.db`  
- 这是**完整快照**，可单独拷贝到其他电脑，**无需**附带 `cat.db-wal` / `cat.db-shm`。  
- API 运行时也可执行（比直接复制 `cat.db` 更安全）。  
- 若证书表格中有图片，还需一并复制整个 **`data/uploads/`** 目录。

手动备份（停 API 后）也可以：

```powershell
# 先 Ctrl+C 停掉 npm run dev:local / pm2
Copy-Item data\cat.db data\backups\cat-backup-手动.db
Copy-Item -Recurse data\uploads data\backups\uploads-手动   # 若有图片
```

#### 恢复数据

**方式 A：后台界面（超级管理员）**

**设置 → 数据维护** →「选择备份文件恢复」。恢复前会自动在 `data/backups/` 保留当前库副本。

**方式 B：命令行**

```powershell
# 先停 API，再执行（将路径换成你的备份文件）
npm run restore:data -- data\backups\cat-backup-2026-06-01_191154.db

npm run dev:local
```

也可手动恢复：

```powershell
# 1. 停 API
Remove-Item data\cat.db-shm, data\cat.db-wal -ErrorAction SilentlyContinue
Copy-Item data\backups\你的备份.db data\cat.db -Force
# 2. 重启 npm run dev:local
```

#### 清理未使用的上传图片

**设置 → 数据维护** →「扫描未使用文件」→「清理未使用图片」。会扫描数据库中所有 `/uploads/…` 引用，删除 `data/uploads/` 里未被使用的文件。

#### 自动备份（定时）

**设置 → 数据维护 → 自动备份**（超级管理员）：

- 勾选「启用自动备份」并选择间隔（1 / 6 / 12 / 24 小时或 7 天）
- **保存目录**：项目内相对路径（如 `data/backups`）或绝对路径（如 `D:\cat-backups`）
- **保留份数**：超出后自动删除最旧的 `backupdata-auto-*.db`（0 = 不删）

> 自动备份在 **API 进程运行期间**生效；停止 `npm run dev:local` / PM2 后不会继续计时。服务重启后若已到期会立即补跑一次。

> **版本升级说明（2026-05 起）**  
> - 新功能会在 API **首次启动时自动迁移** SQLite（新增 `table_templates` 表、`certificates.column_order` 字段），**无需手工执行 SQL**。  
> - 旧证书若从未保存过列顺序，`column_order` 为空，打开后仍按内置默认列顺序显示；在编辑页拖拽列并 **保存** 一次后即写入数据库。  
> - **表格模板库不会自动写入默认列**；若升级前已有自动种子的「标准血统表」，可在 **设置 → 表格模板库** 中删除。

### 9.2 上传目录 `data/uploads/`

表格单元格中的图片路径形如 `/uploads/xxx.png`，文件实体在此目录。  
迁移后若图片不显示，检查该目录是否一并拷贝。

### 9.3 布局 JSON

| 文件 | 作用 |
|------|------|
| `layout-settings.json` | 编辑框位置、列绑定、`fontSourceId` 等 |
| `public/layout-settings.json` | 开发时与根目录同步 |
| `src/default-layout-settings.json` | 「设为默认」后的出厂布局 |

开发时改布局会写回根目录与 `public/`。  
证书内布局以 **数据库中 `layout_overrides`** 为准；JSON 多用于本地备份或与预设同步。

### 9.4 根目录 SVG 文件

如 `模板260528.svg` 等为本地参考文件；**线上使用的模板以数据库 `svg_templates` 为准**（在 templates 页面上传）。

### 9.5 表格列与表格模板

| 内容 | 说明 |
|------|------|
| 内置默认列 | `src/fieldMap.js` 中的 `FIELD_MAP` / `COLUMNS`（血统证书标准列，约 35 列） |
| 证书内列顺序 | 数据库 `certificates.column_order`（JSON 数组）；编辑页拖拽列后需 **保存证书** |
| 表格模板库 | 数据库 `table_templates`；后台 **设置 → 表格模板库**，或编辑页工具栏 **表格模板** |
| 与布局预设区别 | **布局预设** 存编辑框位置/字体；**表格模板** 只存 **列名与列顺序**，不含单元格数据 |

新建证书时可从表格模板快速套用列结构；具体数据仍在 `certificate_rows` 中。

## 10. 字体源与编辑框字体

1. 登录后打开 **字体源**：http://localhost:5173/fonts.html  
2. 添加字体 URL（CDN 或本机可访问路径），勾选 **启用**，保存。  
3. 在编辑页选中 **编辑框**，右侧布局工具栏 **「字体」** 下拉选择（列表仅显示已启用项）。  
4. 每个编辑框的字体保存在 `layoutOverrides` 的 `fontSourceId` 字段中。  
5. 修改字体源后若预览未更新，**刷新页面** 或重启 `dev:local`。

可选构建变量（`npm run build` 时）：

```env
# .env 中可选
# VITE_FONT_URL=https://你的CDN/xxx.ttf
# VITE_FONT_URL=/font/SourceHanSansCN-Medium.ttf
```

- 服务器上 **不要** 拷贝开发用 `node_modules`，应在服务器执行 `npm install` 与 `npm run build`。  
- 部署包需保留或迁移 **`data/cat.db`** 与 **`.env`**。

### 12.1 发布新版本到已有服务器（重要）

每次发版到线上，建议按此顺序操作：

1. **备份** `data/cat.db` 和 `.env`（以及 `data/uploads/` 若用过图片列）。
2. 上传/覆盖新版本源码（**保留** `data/`、`.env`，**不要** 用 Windows 的 `node_modules` 覆盖 Linux）。
3. SSH 进入网站目录，执行：

```bash
cd /www/wwwroot/你的站点目录
npm install          # 若 package.json 或原生依赖有变
npm rebuild better-sqlite3
npm run build        # 必须：前端 dist 含表格模板库等新页面
pm2 restart cat-svg  # 必须：加载新 server 代码与数据库迁移
```

4. **验证**（浏览器或 curl）：
   - `https://你的域名/api/meta` → `features` 中应含 **`table_templates`**
   - 登录后台 → **设置 → 表格模板库** 可新建/编辑空模板（不再自动创建默认模板）
   - 打开一张证书，拖拽列顺序 → **保存** → 刷新，顺序应保持不变

> **说明**  
> - `VITE_FONT_URL` 仅在 **`npm run build` 打包前端** 时生效；服务器运行 `server/index.js` **不需要** 配置该变量。  
> - 若 PM2 日志出现 `VITE_FONT_URL` 或 API 无法启动，说明 `server/` 未更新到最新版，请重新上传并 `pm2 restart`。  
> - 仅覆盖 `dist/` 而不重启 API，会出现前端有新功能、接口 404 的情况；仅重启 API 而不 `build`，会出现后台缺少「表格模板库」等界面。

## 13. 常见问题

### Q1：`crypto.getRandomValues is not a function` 或 Node 版本报错

**原因**：Node 版本过低。  
**处理**：安装 Node 20.19+ 或 22.12+，新开终端后 `node -v` 再执行 `npm run dev:local`。

### Q2：`better-sqlite3` 安装 / 加载失败

**原因**：原生模块与当前 Node/系统不匹配。  
**处理**：

```powershell
Remove-Item -Recurse -Force node_modules
npm install
```

Windows 仍失败时，安装「Visual Studio Build Tools」中的 **C++ 桌面开发**，或安装 https://github.com/nodejs/node-gyp#on-windows 所需组件后重试。

### Q3：能打开页面但登录失败

- 确认 API 已启动（终端有 `[CAT API] http://localhost:3001`）。  
- 确认 `.env` 中密码与输入一致；改过密码则 `npm run reset:admin`。  
- 若 `data/cat.db` 来自旧机，用户名以库中为准（一般为 `admin`）。

### Q4：图片 / 字体不显示

- 图片：检查 `data/uploads/` 是否拷贝。  
- 字体：/fonts.html 中源是否 **启用**、URL 在新电脑网络下是否可访问。  
- 开发时确认用的是 `dev:local`（同时有 API），不是单独 `dev:web`。

### Q5：提示「3001 为旧版 API」或表格模板 / 图片上传 404

**处理**：Ctrl+C 停止后执行 `npm run dev:local`（会自动 `kill-stale-api`），或手动结束占用 3001 的进程。  
服务器上执行 `pm2 restart cat-svg`，并确认已 `npm run build` 且 `/api/meta` 的 `features` 含 `table_templates`。

### Q6：API 启动报错 `VITE_FONT_URL` / `import.meta.env`

**原因**：旧版 `server/index.js` 错误引入了前端模块。  
**处理**：更新到最新代码（含 `src/fieldMap.js`），重启 API。开发环境执行 `npm run dev:local`。

### Q7：表格有数据但 SVG 无文字

- 检查是否配置了 **编辑框** 并与列绑定。  
- 检查当前证书使用的 **SVG 模板** 是否在 templates 页存在。  
- 布局是否保存在当前证书的 `layout_overrides` 或布局预设中。

### Q8：拖拽列顺序保存后刷新又乱序

**原因**：旧版未写入 `column_order`，或 API 未更新。  
**处理**：确认 `/api/meta` 含 `table_templates`；拖拽后点击 **保存**；升级后旧证书需重新保存一次列顺序。

### Q9：从 zip 恢复后 `npm install` 很慢

正常现象；`node_modules` 较大。只需安装一次。若 zip 已含同架构 `node_modules` 且能正常启动，可暂不重装。

## 14. 配置完成检查清单

在新电脑上逐项确认：

- [ ] `node -v` ≥ 20.19（或 22.12+）
